# Inclusive Urban Sim

포용적 주민참여를 위한 LLM 다중 에이전트 시뮬레이션: 기술적 구현에 따른 효과 비교

```
최근수정일: 26. 02. 19. 14:34
작성자: 인하대학교 공간정보공학과 도시공간연구실 이태수
연구 관련 문의: dosiri24@inha.edu
```

## 연구 개요

LLM 에이전트들이 도시계획 주민참여를 시뮬레이션하는 연구입니다. 기존 연구들은 단일 모델만 사용해서 의견 다양성이 부족하다는 한계가 있었는데, 이 연구에서는 **컨텍스트 분리**, **복수 모델 사용**, **사회자 에이전트 도입** 등 기술적 구현 방식을 바꿔가며 **사회적 약자 의견 반영도**에 어떤 영향을 주는지 비교합니다. 이에 따라 제가 정의한 연구 질문은 아래와 같습니다. 

### 연구 질문

1. LLM의 컨텍스트를 분리하면 토론이 더 다양해지는가?
2. 여러 LLM 모델을 섞어 쓰면 포용성이 증가하는가?
3. 사회자 에이전트가 사회적 약자의 발언 기회를 보장하는가?

## 실험 설계

### 본 실험 (가설 검증)

| 레벨 | 세션 | 모델 | 사회자 | 가설 | 검증 목적 |
|------|------|------|--------|------|-----------|
| **Lv.1** | 단일 | Gemini | X | - | 선행 연구 재현, 한계 확인 |
| **Lv.2** | 개별 | Gemini | X | H1 | 컨텍스트 분리 효과 |
| **Lv.3** | 개별 | 복수(5개) | X | H2 | 모델 다양화 효과 |
| **Lv.4** | 개별 | 복수(5개) | O | H3 | 사회자 역할 효과 |

**규모**: 120회 토론 (30세트 × 4레벨)

Lv.1에서 Lv.4로 갈수록 기술적 요소가 추가됩니다. 절제연구 방법론에서 영감을 받아 이를 역순으로 설계한 것인데, 각 레벨에서 포용성 지표 변화를 관찰하려 합니다. 

## 연구의 특징

이 연구에서 제가 중요하게 보는 포인트는 포용성을 관찰 가능한 구조로 만들고, 그 변화의 비교가 가능하도록 실험을 설계했다는 점입니다. 우선 레벨의 변화는 같은 토론 의제와 흐름을 유지한 채로 구현 요소만 바꿉니다. 컨텍스트를 나누고, 모델 다양성을 부여하고, 사회자 에이전트의 개입을 추가하고. 이렇게 하면 어떤 기술적 변화가 실제로 포용성 지표 및 토론 품질에 영향을 준 것인지, 최소한의 비교 가능성을 확보할 수 있다고 생각했습니다.

이 시스템은 토론에서 에이전트가 상대방의 발언과 별도로 스스로의 속마음(Think)를 누적하고, 그 생각이 다시 다음 발언 및 Think에 영향을 주는 흐름으로 설계했습니다. 결과적으로는 에이전트가 토론에서 어떤 말을 했는지뿐 아니라 왜 그렇게 생각했는지도 추적할 수 있습니다. 실제로 시뮬레이션 수행을 끝내고 분석을 진행하면서 정성분석에서도 유의미한 패턴을 관찰할 수 있었습니다.

상호작용의 정량적 분석을 가능하게 하기 위해 **지목 시스템**을 넣었습니다. 발언시 필요한 경우 공감/비판/인용/질문을 명시적으로 남기게 하였습니다. 또한 각 에이전트가 최종 합의문을 읽고 투표하게 만들었습니다. 토론의 상호작용뿐 아니라 결과적인 에이전트들의 만족도까지 한 흐름 안에서 다루게 됩니다. 


## 프롬프트 설계

### 1. Role Playing
 
  에이전트에게는 고유한 주민 특성과 역할이 부여됩니다. "You are not an AI"를 명시하여 역할극 성격을 강화하고 LLM이 페르소나에 몰입하도록 설계했습니다. 이때 저는 주민에게 재개발에 대한 찬/반 혹은 만족/불만족 입장을 미리 주지 않았습니다. 각 에이전트가 토의 과정에서 정보를 듣고 갈등을 겪으면서 **스스로 입장을 형성**하도록 구성했습니다. 연구의 핵심이라고 생각합니다.

### 2. 독립접 판단 유지

 초기 Pilot test에서 토론이 지나치게 한쪽으로 수렴하는 문제가 반복되어, 프롬프트에 Independent Judgment Rules를 명시했습니다. “남이 뭐라고 하든 내 상황과 이해관계 기준으로 판단하고, 자연스럽게 반대/갈등이 나오는 것이 정상”이라는 규칙을 넣어서, 형식적 합의나 분위기 추종을 줄이려 했습니다.

### 3. 지식 수준 제한 (주민다운 발화 유지)

 주민은 전문가가 아니기 때문에, 모르는 용어나 숫자는 모른다고 말해도 되고, 아는 범위에서 자신의 경험 언어로 해석하도록 제한합니다. 이 제한이 없으면 모든 에이전트가 지나치게 논리적인 분석가스러운 말투로 수렴해버려 토론 다양성이 줄어듭니다.

### 4. 내면 사고 분리

  모든 에이전트는 자신의 발화와 별도로 자신의 속마음을 누적하여 저장하고 이를 바탕으로 성찰합니다. 상대방이 발언할 때 마다 왜 그렇게 말했는지, 나는 그거에 대해 어떻게 생각하는지 기록합니다. 

### 5. 지목 시스템

 발화 시 누구의 의견에 반응했는지 명시적으로 기록합니다. 공감, 비판, 인용, 질문을 통해 에이전트 간 의견 교환을 정량적으로 분석할 수 있습니다.

## 페르소나 설계

 에이전트는 찬성/반대 입장을 사전에 부여받지 않습니다. 인구통계학적 속성만으로 페르소나가 구성되며, 이를 바탕으로 스스로 입장을 형성합니다. 각 에이전트가 부여 받는 속성은 아래와 같습니다. 

```
연령대, 성별, 직업, 주거유형, 주거 소유 형태, 연소득, 거주기간, 가구구성, 재개발 지식
```
이 속성은 인천시~주안2동을 범위로 2020~2025년 사이 발표된 국가 통계와 마이크로데이터를 바탕으로 비율을 계산했습니다.

## 대상지

**인천광역시 미추홀구 주안2동**

이 곳은 주안 4동과 함께 전체 지역이 재정비촉진지구로 지정되어 있습니다. 재개발이라는 한국 고유의 도시 의제에서 관련 이해관계 충돌 현안을 다수 관찰할 수 있습니다. 선정한 대상지를 토대로 사전에 설정한 토론 의제는 아래와 같습니다. 

```
미추5구역 촉진계획 세부안에 대한 토의 및 개선 방향 도출
```

## 레포지토리 내용

### 폴더 구조

```
llm_api/        # LLM API 래퍼 (gemini, chatgpt, claude, kimi, exaone)
agent_api/      # 에이전트 시스템 (Agent, Memory, PromptBuilder)
debate/         # 토론 시뮬레이션 (DebateSimulation, 페르소나 로더, 응답 파서)
prompts/        # 프롬프트 파일 (시스템 가이드, 토론 규칙, 지역 맥락, 취약계층 페르소나)
datas/          # 통계 데이터 (인구, 주택, 소득 등)(공유 불가로 인해 GitHub에는 비공개)
logger/         # 로깅 시스템
outputs/        # 실험 결과 저장
analysis/       # 정량 분석 스크립트
tools/          # 유틸리티 스크립트
```

### 주요 파일

`main.py`는 시뮬레이션의 실행 진입점입니다. 제가 연구에서 수행한 에이전트 생성부터 토론 진행까지, 전체 흐름을 확인할 수 있습니다.

`api_price_predict.py`는 각 실험 레벨별 LLM API 비용을 사전에 예측합니다. 연구계획서 작성 단계에서 연구비 추정을 위해 작성하였으나, 혹시 있을 후속 연구자를 위해 Github에 함께 공개합니다. 다만 저의 경우 해당 코드가 캐싱을 과도하게 예측한 탓에 당초 예측 비용보다 더 사용하였습니다.

`.env.template`는 API 키와 실험 파라미터 설정을 위한 템플릿입니다. 이 파일을 `.env`로 복사한 뒤 본인의 API 키를 입력하면 됩니다.

`debate/config.py`는 페르소나 생성에 사용되는 속성별 비율과 시뮬레이션 설정값을 담고 있습니다. 각 비율은 인천시~주안2동 범위의 국가통계를 기반으로 산출되었으며, 다른 지역에 적용할 경우 해당 지역 통계 교체해야 합니다.

`prompts/debate_rule.md`는 토론 의제와 라운드별 진행 규칙을 정의합니다. 실제 미추5구역 재개발 계획안 데이터(비례율, 세대수, 분양가 등)가 포함되어 있으므로, 다른 의제를 다룰 경우 이 파일을 수정해야 합니다.

`prompts/local_context.md`는 대상지의 현황과 재개발 진행 경과를 담고 있습니다. 에이전트가 지역 맥락을 이해하는 데 사용되므로, 대상지가 바뀌면 해당 지역 정보로 교체해야 합니다.

### 시스템 산출물 구조

실행 결과는 클래스 호출 시 지정한 경로에 저장됩니다. 기본적으로는 토의 기록(debate), 내면 생각(think), 토큰 사용량(tokens), 에이전트 목록(agents), 최종 합의문(consensus)이 각각 파일로 분리되어 저장됩니다. 정량/정성 분석도 이 결과물을 가지고 진행했습니다.
